# 사원 정보 관련

## _dept_
- 부서 코드와 부서 활성 여부 등 부서 기본 정보를 저장하는 엔티티

**속성**

| 필드명             | 비고          | 설명             | 
|-----------------|-------------|----------------|
| `id         `   | INT         |                |    
| `dept_code  `   | varchar(3)  | 부서코드 ex- '001' |    
| `dept_name  `   | varchar(20) | 부서명            |    
| `is_active  `   | bool        | 활성화 여부         |    
| `created_at `   | DATETIME    |                |    
| `updated_at `   | DATETIME    |                |    

**행위**
- `static registerDept()`: 부서 생성: deptCode, deptName
- `activate()`: 부서 활성화
- `deactivate()`: 부서 비활성화
- `renameDept()`: 부서 이름 변경

**규칙**
- 부서 비활성화는 해당 부서에 'ACTIVE' 상태인 사원이 없을 때만 가능하다.
- 신규 부서는 기본적으로 활성 상태이다.
- 비활성화 부서는 일반 조회 대상에서 제외된다.
- 부서코드는 부서 생성 후 변경하지 않는다.
- 부서 정보 수정 / 등록은 SystemRoleCode = `ADMIN`만 할 수 있다.

- 사원 계정 및 인적사항, 입사일 등 사원 기본 정보를 저장하는 엔티티

**속성**

| 필드명                | 비고               | 설명                               |
|--------------------|------------------|----------------------------------|
| `id`               | int              |                                  | 
| `emp_no`           | varchar(9)       | 사번, 202603xxx 형식(입사연월+연번)        | 
| `emp_name`         | VARCHAR(20)      | 직원 이름                            | 
| `login_Id`         | VARCHAR(50)      | 아이디                              | 
| `emp_Password`     | varchar(100)     | 비밀번호                             | 
| `email`            | VO / varchar(50) | 직원의 사내 이메일(emp_id + @haruon.com) | 
| `extension_no`     | VARCHAR(8)       | 사무실 직통번호                         | 
| `status`           | Enum             | 재직상태                             | 
| `system_Roles`     | SET<ENUM>        | 시스템 권한                           | 
| `hire_at`          | DATE             | 입사일                              | 
| `resign_at`        | DATE             | 퇴직일                              | 
| `emp_Files`        | List             | 사원 프로필사진 / 전자서명 이미지              | 
| `emp_Belongings`   | List             | 사원 소속                            | 
| `created_at `      | DATETIME         |                                  |    
| `updated_at `      | DATETIME         |                                  |    


#### **Emp_Status**
_Enum_
**상수**
- `PENDING` : 승인보류(인사과 가입 승인 전)
- `ACTIVE` : 재직중
- `RESIGNED` : 퇴직
- `SUSPENDED` : 장기휴직 등 퇴직하지 않았지만 재직이 불가능한 상태

#### **SystemRoleCode**
_Enum - ElementCollection_                 
**상수**
- `EMPLOYEE`: 일반권한
- `FRANCHISE`: 프랜차이즈 관리 권한
- `IT`: 시스템 부서 권한
- `HR`: 인사과 권한
- `DEPT_MANAGER`: 부서 관리자 권한
- `ADMIN` : 시스템 총괄 권한

**행위**
- `static register()`: 사원 등록 : EmpRegisterRequest(emp_no, empName, emp_id, rawPassword)
- `approveRegister()`: 가입승인
- `changeResignedEmpInfoByAdmin()`: 퇴직자정보 수정
- `changeEmpFile()` : 파일 추가(변경)
- `changeFileActiveStatus()` : 파일 활성화/비활성화
- `removeFile()`: 파일 삭제
- `changeBelongingsByAdmin()` : 사원의 소속정보 수정(추가)
- `changeInfoBySelf()`: 본인 정보 수정
- `changeInfoByDeptManager()`: 부서관리자용 사원 정보 수정
- `changeInfoByAdmin()`: 총괄 관리자용 사원 정보 수정

**규칙**
1. 계정 및 인증
   - 비밀번호는 영문 + 숫자 + 특수문자 포함 8자 이상이어야 한다.
   - 비밀번호는 반드시 해시(암호화)하여 저장한다.
   - 비밀번호 변경 시:
   - 기존 비밀번호와 동일한 값으로 변경할 수 없다.
   - 본인 수정 시 현재 비밀번호 검증이 필수이다.
2. 식별값 및 중복
   - 사번(empNo)과 아이디(loginId)는 중복될 수 없다.- 비밀번호는 [영문+특수문자+숫자]포함 8자 이상이여야 하며, 해시를 만들어서 저장한다.   
   - 이메일은 시스템에서 생성되며 다음 형식을 따른다:- 사내 이메일 주소는 『아이디 @ 도메인』 형식이다.
   - loginId@companyDomain
   - 내선번호는 『000-0000』 형식이다
   - 이메일은 유니크해야 한다.
   - 아이디와 사번은 중복을 허용하지 않는다.
3. 기본 생성 규칙
   - 사원 신규 생성 시:
   - 상태는 PENDING으로 설정한다.
   - 비밀번호는 암호화된 값으로 저장한다.
4. 상태 전이 규칙
   - `PENDING` → `ACTIVE` : 가입 승인 시에만 가능 (`PENDING` 상태에서만 가입 승인 가능하다.)
   - `ACTIVE` → `RESIGNED` : 퇴직 처리 시 변경
   - `ACTIVE` 상태에서만:
     - 개인정보 수정 가능
     - 파일 변경 가능
     - 소속 변경 가능
5. 퇴직 처리 규칙
   - 퇴직일 입력 시: `status` = RESIGNED로 변경된다.
   - 모든 소속 정보(`EmpBelongings`)의 `end_at`은 퇴직일로 설정된다.
   - 퇴직일은 입사일 이후여야 한다.
6. 개인정보 수정 규칙
   - 모든 수정은 `ACTIVE` 상태에서만 가능하다
   - 본인 수정
      - 현재 비밀번호 검증 필수
      - 변경 가능 항목: 내선번호, 비밀번호
   - 부서 관리자 수정
      - 변경 가능:
      - 내선번호
      - 시스템 권한 (본인보다 낮은 등급만)
   - 총괄 관리자/HR 수정
      - 변경 가능: 이름, 비밀번호, 내선번호, 상태, 시스템 권한, 입사일
7. 내선번호 규칙
   - 형식: 000-0000
   - 선택 입력 값 (nullable)
8. 파일 및 부가정보
   - 동일 파일 타입은 하나만 활성화 가능
   - 새로운 파일 등록 시 기존 파일은 비활성화됨
   - 모든 사원의 파일 수정은 `ACTIVE` 상태에서만 가능하다
9. 소속 정보 규칙
   - 소속 등록 시:
      - 부서, 직급, 시작일, primary 여부 필수
      - primary 소속은 하나만 존재 가능
      - 퇴직 시 자동 종료 처리됨
   - 모든 사원의 소속 수정은 `ACTIVE` 상태에서만 가능하다

## _emp_file_
- 사원 프로필 사진, 서명파일 등 사원 관련 파일 정보를 저장하는 엔티티

**속성**

| 필드명              | 비고            | 설명                                              |
|------------------|---------------|-------------------------------------------------|
| `id`             | int           |                                                 | 
| `emp_id`         | int           | EMP.id                                          | 
| `original_name ` | VARCHAR(200)  | 원본명                                             |
| `stored_name   ` | VARCHAR(200)  | 서버 저장파일명(UUID)                                  |
| `mime_type     ` | VARCHAR(100)  | image/jpeg, image/png 등                         |
| `extension     ` | VARCHAR(20)   | 확장자(jpg, png 등)                                 |
| `file_size     ` | BIGINT        | 바이트 단위, 5MB(5 * 1024 * 1024 = 5,242,880B) 미만 허용 |
| `file_type`      | Enum          | 파일 종류                                           |
| `is_active     ` | bool          | 활성화여부                                           |
| `created_at `    | DATETIME      |                                                 |    
| `updated_at `    | DATETIME      |                                                 |    

### **FileType**
_Enum_                                               
**상수**
- `SIGNATURE` : 전자서명
- `PROFILE_PICTURE` : 사원프로필사진

**행위**
- `static addFile()`: 파일(프로필사진, 서명 이미지) 등록 - EmpFileRequest

**규칙**
- 파일을 추가하면 추가한 파일은 `ACTIVE`상태가 되며, 같은 타입의 파일을 비활성화한다.
- 확장자(`extension`)은 "jpg", "jpeg", "png", "gif"만 허용한다
- MIME 타입(`mime_type`)은 "image/jpg", "image/jpeg", "image/png", "image/gif"만 허용한다.
- 파일 사이즈는 5MB를 초과할 수 없다. 
- 파일 메타정보(원본파일명, 확장자, 파일크기, mimeType)은 필수 값이다.
- 소속된 사원 정보는 필수 값이다.
- 저장 파일명은 시스템이 생성한다. 예: UUID

## _attendance_
- 사원의 출퇴근/근태 상태와 승인 정보를 저장하는 엔티티
- 특정 직원의 특정 날짜 근태 기록 1건을 나타낸다.

**속성**

| 필드명                | 비고             | 설명          |
|--------------------|----------------|-------------|
| `id              ` | INT            |             |
| `emp_id          ` | INT            | 근태 기록 대상 사원 |
| `attendance_state` | ENUM           | 근태 상태       |
| `attendance_date`  | Date           | 근태 일자       |
| `start_at        ` | TIME           | 시작 시간       |
| `end_At        `   | TIME           | 종료 시간       |
| `approved_at     ` | DATETIME       | 승인 시각       |
| `approved_by     ` | INT            | 승인자         |
| `edited_at     `   | DATETIME       | 수정 시각       |
| `edited_By     `   | INT            | 수정한 사람      |
| `edit_reason     ` | varchar(100)   | 수정사유        |
| `created_at `      | DATETIME       |             |    
| `updated_at `      | DATETIME       |             |  

**AttendanceStatus**

_Enum 상수_
- `NORMAL` : 정상근무
- `HALF_DAY_LEAVE` : 반차
- `ALL_DAY_LEAVE` : 연차
- `SICK_LEAVE` : 병가
- `ABSENT` : 결근
- `LATE_EARLY` : 지각 및 조퇴

**행위**
- `static registerAttendanceByEmp()` : 직원이 출근 찍을 때 근태 데이터 생성 
- `recordEndAtByEmp()` : 직원이 퇴근을 찍을 때 기존 근태 데이터 수정
- `approveAttendance()`  : 부서 시스템 관리자가 근태 확정
- `changeAttendanceByDeptManager()` : 부서 시스템 관리자가 근태 수정
- `markEditor()` : 수정용 메서드, 편집정보 기록
- `registerAttendance()` : 전날 근태 마감, 마감용 객체 생성 메서드
- `changeAttendanceTime()` : 마감용 메서드, 근태 마감시 시간 조정
- `changeAttendanceStatus()` : 마감용 메서드, 근태 상태 조정

**규칙**
1. 근태 기본 규칙
   - 근태는 emp와 attendanceDate를 기준으로 특정된다.
   - 1일 1근태가 원칙이다.
   - 다만 반차 등 시간 분할 근무가 필요한 경우 같은 날짜에 추가 근태가 생성될 수 있다.
   - 이 경우에도 하루 근태는 업무 규칙상 최대 2건까지 허용한다.
2. 근태 생성 규칙
   - 직원 출근 기록으로 생성 : 직원이 출근을 기록하면 근태가 생성된다.
     - 생성 시 필수값: `emp`, `attendanceDate`, `startAt`
   - 관리자/마감 처리에 의한 생성 : 관리자가 직접 근태를 등록하거나, 마감 로직이 근태를 생성할 수 있다.
     -  필수값: `emp`, `attendanceDate`, `attendanceStatus` [`NORMAL`, `LATE_EARLY`, `HALF_DAY_LEAVE` 상태는 시간 정보(startAt, endAt)가 반드시 필요하다.]
3. 출퇴근 기록 규칙
   - 출근 시 근태가 생성된다.
   - 출근 시각은 근태일자의 시각이어야 한다.
   - 퇴근 기록
     - 출근 기록이 있는 경우에만 퇴근을 기록할 수 있다.
     - 퇴근 시각은 null일 수 없다.
     - 퇴근 기록은 같은 날짜의 근태에 대해서만 가능하다.
     - 퇴근 시각은 출근 시각보다 빠를 수 없다.
4. 근태 상태 규칙
   - 시간/종료 시각 필수 상태 : `NORMAL`, `LATE_EARLY`, `HALF_DAY_LEAVE`
   - 시간 비포함 상태: `ALL_DAY_LEAVE`, `SICK_LEAVE`, `ABSENT`
5. 근태 승인
   - 근태는 상태가 존재할 때만 승인할 수 있다.
   - 같은 날짜의 근태는 당일 승인할 수 없다.
   - 이미 승인된 근태는 다시 승인할 수 없다.
   - 승인 시 기록되는 값: `approvedBy`, `approvedAt`
6. 근태 수정
   - 부서 관리자는 부서 내 직원의 근태를 수정할 수 있다.
   - 부서 관리자는 출퇴근시간만 변경이 가능하며, 상태를 직접적으로 수정할 수 없다.
   - 이미 승인된 근태는 수정할 수 없다.
   - 수정 시 필수값: `editAt`, `editReason`, `editedBy`, `start_at 또는 end_at`
   - 수정 사유는 공백일 수 없다.
   - 수정할 내용이 실제로 존재해야 한다.
   - 수정 가능 항목: `출근 시각`, `퇴근 시각`
   - 근태가 수정되면 아래 정보를 남긴다: (수정자) `editedBy`, (수정일시) `editedAt`, (수정사유) `editReason`

## _Emp_Leave_
- 특정 사원, 연도의 모든 종류의 휴가 보유/사용/잔여 현황 전체를 관리

**속성**

| 필드명                        | 비고               | 설명         | 
|----------------------------|------------------|------------|
| `id         `              | INT              |            |    
| `emp_id     `              | INT              | `Emp.id`   |    
| `grant_year `              | INT              | 대상연도       |    
| `annualBaseGrantDays `     | DECIMAL(4,1)     | 총 연차 수     |    
| `annualUsedDays  `         | DECIMAL(4,1)     | 연차 사용 일수   |    
| `specialGrantDays  `       | DECIMAL(4,1)     | 특별휴가 부여 수  |    
| `specialUsedDays  `        | DECIMAL(4,1)     | 특별휴가 사용 일수 |

- 이 엔티티는 연차, 특별휴가, 대체휴무의 부여일수/사용일수/잔여일수를 관리한다.

- 각 휴가 유형별 부여일수는 음수가 될 수 없다.
- 휴가 사용일수 차감은 LeaveDraft가 결재완료 상태가 된 이후에만 수행한다.
- Admin은 각 휴가 유형의 부여일수를 수동 조정할 수 있다.


**연차 산정 규칙**
```
- 연차는 회사 정책에 따라 매년 1월 1일 기준으로 부여한다.
- 입사연도:
  - 입사일 기준 근무 개월 수를 기준으로 월할 계산하여 부여한다.
  - 1개월 개근 시 1일 발생하며, 최대 11일까지 부여한다.
- 입사연도 이후:
  - 매년 1월 1일에 기본 15일을 부여한다.
  - 이후 근속 2년마다 1일씩 가산하며, 최대 25일까지 부여한다.
- 근속연수는 입사일을 기준으로 산정한다.
```

## _Approval_
- Approval은 하나의 Draft에 대한 결재 흐름을 관리하는 애그리거트
- 결재선(Approver)을 보유하며, 상신 전/상신 후/진행 중/완료/반려 상태를 관리한다.

**속성**

| 필드명                  | 비고          | 설명              |
|----------------------|-------------|-----------------|
| `id `                | int         |                 |
| `draft_id `          | int         | draft.id        |
| `status `            | varchar(10) | Enum, 결재 상태     |
| `approvers `         | List        | Approver.결재자 목록 |
| `created_at        ` | datetime    |                 |  
| `updated_at        ` | datetime    |                 |   

#### **ApprovalStatus**
_Enum_
**상수**
- `UNSUBMITTED` : 임시저장 상태, 미상신
- `WAITING` : 결재대기
- `IN_PROGRESS` : 결재진행중
- `APPROVED` : 결재완료
- `REJECTED` : 결재반려

**행위**
- `createDraft()` : 임시저장된 기안서대상 임시 승인정보 엔티티 생성.
- `createSubmitted()` : 기안 상신한 기안서대상 승인정보 엔티티 생성.
- `approve()` : 해당 차례의 결재자의 승인
- `reject()` : 해당 차례의 결재자의 반려처리

**규칙**
1. 엔티티 생성 규칙
   - Approval은 반드시 하나의 Draft와 1:1로 연결된다.
   - 생성 시 상태는 항상 WAITING으로 시작한다.
   - 연관
     - Draft생성 시, Approval가 1:1 관계로 같이 생성되며, 결재선에 따라 Approver이 1:N관계로 생성된다.
2. 결재자 추가 
   - 결재자 추가 규칙 (addApprovers)
   - 동일한 Emp는 중복으로 결재자에 등록될 수 없다.
   - 동일한 결재 순서(order)는 중복될 수 없다.
   - 결재자는 최소 1명 이상 등록되어야 정상적인 결재 진행이 가능하다.
   - 결재자 추가는 결재 진행 전에만 이루어져야 한다
3. 결재상태 규칙
   - 상태 전이 흐름
     ```
     (UNSUBMITTED -> ) WAITING -> IN_PROGRESS -> APPROVED
                                  \-> REJECTED
     ```
4. 생성 : 엔티티 생성시 기안 제출전후에 따라 생성 규칙이 다르다.
   - `createDraft` : 생성 시 상태는 `Unsubmitted` 이다.
   - `createSubmitted` : 생성 시 상태는 `WAITING` 이다
   - 생성시, 필수 파라미터인 결재선은 record인 `List<ApproversParam>`으로 받는다.
   - 결재자 목록(params)은 null일 수 없고 비어 있을 수 없다.
     #### **ApproversParam**
     ```
     - 한 사람은 같은 결재문서에서 한 번만 등장할 수 있다
     - 다만, 본인기안 본인결재의 경우 기안자와 결재자가 동일할 수 있다.
     - 하나의 순서에는 한 명만 배정될 수 있다.
     ```
5. 상신 : 상신은 UNSUBMITTED 상태에서만 가능하다.
   - 결재자가 한 명 이상 존재해야 한다.
   - 상신되면 상태는 WAITING이 된다.
6. 승인 : 승인 후 남은 미처리 결재자가 있으면 상태는 `IN_PROGRESS` 없으면 `APPROVED`
   - 현재 승인 가능한 사람은 미처리 상태(pending)인 결재자 중 가장 낮은 order를 가진 사람이다.
   - 현재 차례의 결재자가 아니면 승인할 수 없다.
   - `REJECT` 또는 `APPROVED` 상태의 결재는 승인할 수 없다.
7. 반려 : 반려 시, 결재 상태는 즉시 `REJECTED`가 된다.
   - 현재 차례(order)인 결재자만 반려를 할 수 있다.

- `moveRecipientToTrash()` - 수신자만 / isSent
- `restoreRecipientFromTrash()` - 수신자만 / isSent

- `markAsRead()` - 수신자만 / isSent
- `isDraft()`

**규칙**
- 발신자, 제목, 내용는 필수다.
- 발송자는 nullable = false, updatable = false로 설정.
- 발송 전은 `sent_at == null`, 발송 후는 `sent_at != null`로 구분한다.  
- 발송 전, 발송 후 여부 상관없이, `Message_sender` 엔티티가 생성된다.
- 발송 전, 발송 후 여부 상관없이, 수신자를 설정하면, `Message_recipient` 엔티티는 수신자 별로 생성된다.
- 발송 전, 발송 후 여부 상관없이, 파일을 첨부하면 `Message_File` 엔티티가 파일 별로 생성된다.
- 발송 전 상태라면 수신자가 없을 수 있다.
- 발송 전 상태라면, `message_sender`는 변경 불가능 하며, `Message_recipient`, `Message_File`을 추가 삭제할 수 있다.
  - 수신자 목록을 일부 추가/삭제를 하더라도, `message_recipient`전체 목록이 교체된다.
- 발송 전 상태라면, 쪽지를 휴지통(`trashed_at != null`)으로 옳기지 못한다.
- 발송 후 상태라면 수신자는 최소 1명 이상이어야 한다.
- 발송 후 발송을 취소할 수 없다.
- 발송 후 title, content, recipient, file을 수정할 수 없다.
- [삭제정책]
  - 임시저장 상태일때 삭제 : Message 물리삭제
  - 휴지통 : `trashed_at != null` 상태이며, 쪽지함이 아닌, 휴지통에서 해당 메세지를 사용자에게 노출시킨다.
  - 보낸쪽지함/받은쪽지함 삭제 : 사용자(발송인, 수신자) 기준 논리삭제, 데이터는 물리 삭제되지 않는다.
- `sender_id = receiver_id` 인 `MessageRecipient`는 "나에게 쓴 쪽지함"에서 조회된다

## _ChatRoom_

- 채팅방 관련 엔티티

**속성**

| 필드명                | 비고          | 설명             |
|--------------------|-------------|----------------|
| `id `              | long        |                |
| `isGroup `         | bool        | 그룹채팅여부         |
| `owner_id `        | long        | 채팅방만든사원 emp.id |
| `last_message_at ` | datetime    | 마지막 채팅시각       |
| `closed_at `       | datetime    | 참여자가 모두 나간 시각  |
| `created_at `      | datetime    |                |  
| `updated_at `      | datetime    |                |

**행위**
- `static createRoom()`
- `rejoin()`
- `markLastMessageAt()`
- `inviteRoomMember`
- `markMemberLeft()`
- `markBookmarkByMember()`
- `changeLastReadMessageByMember()`
- `isParticipating()` : 특정 사원의 해당 채팅방 참가여부 확인

**규칙**
- owner는 채팅방 생성사원이면서 참여자이다
- 채팅방 `생성`, `초대`, `방나가기` 시, 그룹/종료 상태를 재계산한다 
  - 채팅방 멤버가 2명 초과라면 `isGroup = true`
  - 채팅방 멤버가 2명 이하라면 `isGroup = false`
  - 채팅방 멤버가 0명이라면 `closed_at != null`
  - 생성 이후 채팅방에 참여자가 1명일 수 있다.
- `inviteRoomMember()` 시, 초대된 사원 수락여부 상관없이 바로 `ChatRoomMember.join()`이 호출
- 마지막 메시지가 생성되면 `lastMessageAt`을 갱신한다
- 참여자가 채팅방을 나갈 시, `chatRoomMember.leftAt`에 나가는 일시만 표기하며 List.remove()하지 않는다.
- `close_at` 이후 30일 후 batch로 `ChatRoomMember`, `ChatRoom`, `ChatMessage` 모두 삭제한다.
- 모든 참여자가 퇴장하면 `ChatRoom.closed_at`을 기록한다.
- `closed_at != null`인 채팅방은 batch로 연관된 `ChatRoom`, `ChatRoomMember`, `ChatMessage`를 물리삭제한다.
- 종료(`closed_at`)된 `ChatRoom`에는 새로운 메시지를 보낼 수 없다.
- 메시지는 ChatRoom.sendMessage()를 통해 생성한다.
  - 채팅방이 종료되지 않아야한다.
  - sender는 해당 room의 `ChatRoomMember`(참여자)에 속해야 한다.
  - 보내는 시각이, 발송자 기준 채팅방 입장시각이나, 가장 최근의 채팅 시각보다 늦어야 한다.
  - 메시지 생성 시 ChatRoom.lastMessageAt 갱신
  - ChatRoom < - ChatMessage 단방향 관계로 유지하며, application 레벨에서 MessageRepository로 DB에 생성한다.

## _ChatRoomMember_

- 채팅방 관련 엔티티

**속성**

| 필드명                     | 비고            | 설명              |
|-------------------------|---------------|-----------------|
| `id `                   | long          |                 |
| `room_id `              | long          | chatRoom.id     |
| `room_name `            | varchar(20)   | 커스텀채팅방이름        |
| `member_id `            | long          | emp.id          |
| `last_read_message_id ` | long          | chatMessage.id  |
| `is_bookmark `          | boolean       | 채팅방 즐겨찾기 여부     |
| `joined_at `            | datetime      |                 |
| `left_at `              | datetime      |                 |
| `created_at `           | datetime      |                 |  
| `updated_at `           | datetime      |                 |

**행위**
- `static join()` : 사원이 채팅방 입장(`join()`)시, (`joined_At`)이 기록된다.
- `leaveRoom()` : 사원이 채팅방을 나갈시(`leftRoom()`) (`left_at`)이 기록된다
- `markBookmark()` : 즐겨찾기 등록 여부
- `changeLastReadMessage()` : 마지막으로 읽은 메세지 
- `isParticipating()` : 해팅 채팅방 참가여부
- `updateName()` : 멤버별 채팅방 이름 변경

**규칙**
- 퇴장일시(`left_at`)이 null이면 참여중이다
- 퇴장일시(`left_at`)이 null이 아니라면 퇴장 상태이다.
- 퇴장한 참여자는 메시지를 보낼 수 없다.
- 퇴장한 참여자는 `lastReadMessage`를 갱신할 수 없다.
- `lastReadMessage`는 같은 room의 메시지만 가능하다
- `lastReadMessage`는 joinedAt 이후 메시지만 가능하다
- joinedAt은 leftAt보다 늦을 수 없다.
- 즐겨찾기(`is_bookmark`) 기본값은 false이다.
- 즐겨찾기(`is_bookmark`)는 채팅방별로 설정할 수 있다.
- 채팅방 이름 기본값은 null로 저장되며, view에서는 참여자 소속, 이름이 보여진다.
