## _dept_
- 부서 코드와 부서 활성 여부 등 부서 기본 정보를 저장하는 엔티티

**속성**

| 필드명             | 비고          | 설명             | 
|-----------------|-------------|----------------|
| `id         `   | INT         |                |    
| `dept_code  `   | varchar(3)  | 부서코드 ex- '001' |    
| `dept_name  `   | varchar(20) | 부서명            |    
| `is_active  `   | bool        | 활성화 여부         |    
| `created_at `   | DATETIME    |                |    
| `updated_at `   | DATETIME    |                |    

**행위**
- `static registerDept()`: 부서 생성: deptCode, deptName
- `activate()`: 부서 활성화
- `deactivate()`: 부서 비활성화
- `renameDept()`: 부서 이름 변경

**규칙**
- 부서 비활성화는 해당 부서에 'ACTIVE' 상태인 사원이 없을 때만 가능하다.
- 신규 부서는 기본적으로 활성 상태이다.
- 비활성화 부서는 일반 조회 대상에서 제외된다.
- 부서코드는 부서 생성 후 변경하지 않는다.
- 부서 정보 수정 / 등록은 SystemRoleCode = `ADMIN`만 할 수 있다.

## _emp_
- 사원 계정 및 인적사항, 입사일 등 사원 기본 정보를 저장하는 엔티티

**속성**

| 필드명                | 비고               | 설명                               |
|--------------------|------------------|----------------------------------|
| `id`               | int              |                                  | 
| `emp_no`           | varchar(11)      | 사번, 202603xxx 형식(입사연월+연번)        | 
| `emp_name`         | VARCHAR(20)      | 직원 이름                            | 
| `login_Id`         | VARCHAR(50)      | 아이디                              | 
| `emp_Password`     | varchar(50)      | 비밀번호                             | 
| `email`            | VO / varchar(50) | 직원의 사내 이메일(emp_id + @haruon.com) | 
| `extension_no`     | VARCHAR(20)      | 사무실 직통번호                         | 
| `status`           | Enum             | 재직상태                             | 
| `system_Roles`     | SET<ENUM>        | 시스템 권한                           | 
| `hire_at`          | DATE             | 입사일                              | 
| `resign_at`        | DATE             | 퇴직일                              | 
| `emp_Files`        | List             | 퇴직일                              | 
| `emp_Belongings`   | List             | 퇴직일                              | 
| `created_at `      | DATETIME         |                                  |    
| `updated_at `      | DATETIME         |                                  |    


#### **Emp_Status**
_Enum_
**상수**
- `PENDING` : 승인보류(인사과 가입 승인 전)
- `ACTIVE` : 재직중
- `RESIGNED` : 퇴직
- `SUSPENDED` : 장기휴직 등 퇴직하지 않았지만 재직이 불가능한 상태

#### **SystemRoleCode**
_Enum - ElementCollection_                 
**상수**
- `EMPLOYEE`: 일반권한
- `FRANCHISE`: 프랜차이즈 관리 권한
- `IT`: 시스템 부서 권한
- `HR`: 인사과 권한
- `DEPT_MANAGER`: 부서 관리자 권한
- `ADMIN` : 시스템 총괄 권한

**행위**
- `static register()`: 사원 등록 : EmpRegisterRequest(emp_no, empName, emp_id, rawPassword)
- `approveRegister()`: 가입승인
- `changeResignedEmpInfoByAdmin()`: 퇴직자정보 수정
- `changeEmpFile()` : 파일 추가(변경)
- `changeFileActiveStatus()` : 파일 활성화/비활성화
- `removeFile()`: 파일 삭제
- `changeBelongingsByAdmin()` : 사원의 소속정보 수정(추가)
- `changeInfoBySelf()`: 본인 정보 수정
- `changeInfoByDeptManager()`: 부서관리자용 사원 정보 수정
- `changeInfoByAdmin()`: 총괄 관리자용 사원 정보 수정

**규칙**
1. 계정 및 인증
   - 비밀번호는 영문 + 숫자 + 특수문자 포함 8자 이상이어야 한다.
   - 비밀번호는 반드시 해시(암호화)하여 저장한다.
   - 비밀번호 변경 시:
   - 기존 비밀번호와 동일한 값으로 변경할 수 없다.
   - 본인 수정 시 현재 비밀번호 검증이 필수이다.
2. 식별값 및 중복
   - 사번(empNo)과 아이디(loginId)는 중복될 수 없다.- 비밀번호는 [영문+특수문자+숫자]포함 8자 이상이여야 하며, 해시를 만들어서 저장한다.   
   - 이메일은 시스템에서 생성되며 다음 형식을 따른다:- 사내 이메일 주소는 『아이디 @ 도메인』 형식이다.
   - loginId@companyDomain
   - 내선번호는 『000-0000』 형식이다
   - 이메일은 유니크해야 한다.
   - 아이디와 사번은 중복을 허용하지 않는다.
3. 기본 생성 규칙
   - 사원 신규 생성 시:
   - 상태는 PENDING으로 설정한다.
   - 비밀번호는 암호화된 값으로 저장한다.
4. 상태 전이 규칙
   - `PENDING` → `ACTIVE` : 가입 승인 시에만 가능 (`PENDING` 상태에서만 가입 승인 가능하다.)
   - `ACTIVE` → `RESIGNED` : 퇴직 처리 시 변경
   - `ACTIVE` 상태에서만:
     - 개인정보 수정 가능
     - 파일 변경 가능
     - 소속 변경 가능
5. 퇴직 처리 규칙
   - 퇴직일 입력 시: `status` = RESIGNED로 변경된다.
   - 모든 소속 정보(`EmpBelongings`)의 `end_at`은 퇴직일로 설정된다.
   - 퇴직일은 입사일 이후여야 한다.
6. 개인정보 수정 규칙
   - 모든 수정은 `ACTIVE` 상태에서만 가능하다
   - 본인 수정
      - 현재 비밀번호 검증 필수
      - 변경 가능 항목: 내선번호, 비밀번호
   - 부서 관리자 수정
      - 변경 가능:
      - 내선번호
      - 시스템 권한 (본인보다 낮은 등급만)
   - 총괄 관리자 수정
      - 변경 가능: 이름, 아이디, 이메일, 비밀번호, 내선번호, 상태, 시스템 권한, 입사일
7. 내선번호 규칙
   - 형식: 000-0000
   - 선택 입력 값 (nullable)
8. 파일 및 부가정보
   - 동일 파일 타입은 하나만 활성화 가능
   - 새로운 파일 등록 시 기존 파일은 비활성화됨
9. 소속 정보 규칙
   - 소속 등록 시:
      - 부서, 직급, 시작일, primary 여부 필수
      - primary 소속은 하나만 존재 가능
      - 퇴직 시 자동 종료 처리됨

## _emp_file_
- 사원 프로필 사진, 서명파일 등 사원 관련 파일 정보를 저장하는 엔티티

**속성**

| 필드명              | 비고            | 설명                                              |

| `original_name ` | VARCHAR(200)  | 원본명                                             |
| `stored_name   ` | VARCHAR(200)  | 서버 저장파일명(UUID)                                  |
| `mime_type     ` | VARCHAR(100)  | image/jpeg, image/png 등                         |
| `extension     ` | VARCHAR(20)   | 확장자(jpg, png 등)                                 |
| `file_size     ` | BIGINT        | 바이트 단위, 5MB(5 * 1024 * 1024 = 5,242,880B) 미만 허용 |

| `is_active     ` | bool          | 활성화여부                                           |
| `created_at `    | DATETIME      |                                                 |    
| `updated_at `    | DATETIME      |                                                 |    

### **FileType**
_Enum_                                               
**상수**
- `SIGNATURE` : 전자서명
- `PROFILE_PICTURE` : 사원프로필사진

**행위**
- `static addFile()`: 파일(프로필사진, 서명 이미지) 등록 - EmpFileRequest

**규칙**
- 파일을 추가하면 추가한 파일은 `ACTIVE`상태가 되며, 같은 타입의 파일을 비활성화한다.
- 확장자(`extension`)은 "jpg", "jpeg", "png", "gif"만 허용한다
- MIME 타입(`mime_type`)은 "image/jpg", "image/jpeg", "image/png", "image/gif"만 허용한다.
- 파일 사이즈는 5MB를 초과할 수 없다. 
- 파일 메타정보(원본파일명, 확장자, 파일크기, mimeType)은 필수 값이다.
- 소속된 사원 정보는 필수 값이다.
- 저장 파일명은 시스템이 생성한다. 예: UUID

## _attendance_
- 사원의 출퇴근/근태 상태와 승인 정보를 저장하는 엔티티
- 특정 직원의 특정 날짜 근태 기록 1건을 나타낸다.

**속성**

| 필드명                | 비고             | 설명          |
|--------------------|----------------|-------------|
| `id              ` | INT            |             |
| `emp_id          ` | INT            | 근태 기록 대상 사원 |
| `attendance_state` | ENUM           | 근태 상태       |
| `attendance_date`  | Date           | 근태 일자       |
| `start_at        ` | TIME           | 시작 시간       |
| `end_At        `   | TIME           | 종료 시간       |
| `approved_at     ` | DATETIME       | 승인 시각       |
| `approved_by     ` | INT            | 승인자         |
| `edited_at     `   | DATETIME       | 수정 시각       |
| `edited_By     `   | INT            | 수정한 사람      |
| `edit_reason     ` | varchar(100)   | 수정사유        |
| `created_at `      | DATETIME       |             |    
| `updated_at `      | DATETIME       |             |  

**AttendanceStatus**

_Enum 상수_
- `NORMAL` : 정상근무
- `HALF_DAY_LEAVE` : 반차
- `ALL_DAY_LEAVE` : 연차
- `SICK_LEAVE` : 병가
- `ABSENT` : 결근
- `LATE_EARLY` : 지각 및 조퇴

**행위**
- `static registerAttendanceByEmp()` : 직원이 출근 찍을 때 근태 데이터 생성 
- `recordEndAtByEmp()` : 직원이 퇴근을 찍을 때 기존 근태 데이터 수정
- `approveAttendance()`  : 부서 시스템 관리자가 근태 확정
- `changeAttendanceByDeptManager()` : 부서 시스템 관리자가 근태 수정
- `markEditor()` : 수정용 메서드, 편집정보 기록
- `registerAttendance()` : 전날 근태 마감, 마감용 객체 생성 메서드
- `changeAttendanceTime()` : 마감용 메서드, 근태 마감시 시간 조정
- `changeAttendanceStatus()` : 마감용 메서드, 근태 상태 조정

**규칙**
1. 근태 기본 규칙
   - 근태는 emp와 attendanceDate를 기준으로 특정된다.
   - 1일 1근태가 원칙이다.
   - 다만 반차 등 시간 분할 근무가 필요한 경우 같은 날짜에 추가 근태가 생성될 수 있다.
   - 이 경우에도 하루 근태는 업무 규칙상 최대 2건까지 허용한다.
2. 근태 생성 규칙
   - 직원 출근 기록으로 생성 : 직원이 출근을 기록하면 근태가 생성된다.
     - 생성 시 필수값: `emp`, `attendanceDate`, `startAt`
   - 관리자/마감 처리에 의한 생성 : 관리자가 직접 근태를 등록하거나, 마감 로직이 근태를 생성할 수 있다.
     -  필수값: `emp`, `attendanceDate`, `attendanceStatus` [`NORMAL`, `LATE_EARLY`, `HALF_DAY_LEAVE` 상태는 시간 정보(startAt, endAt)가 반드시 필요하다.]
3. 출퇴근 기록 규칙
   - 출근 시 근태가 생성된다.
   - 출근 시각은 근태일자의 시각이어야 한다.
   - 퇴근 기록
     - 출근 기록이 있는 경우에만 퇴근을 기록할 수 있다.
     - 퇴근 시각은 null일 수 없다.
     - 퇴근 기록은 같은 날짜의 근태에 대해서만 가능하다.
     - 퇴근 시각은 출근 시각보다 빠를 수 없다.
4. 근태 상태 규칙
   - 시간/종료 시각 필수 상태 : `NORMAL`, `LATE_EARLY`, `HALF_DAY_LEAVE`
   - 시간 비포함 상태: `ALL_DAY_LEAVE`, `SICK_LEAVE`, `ABSENT`
      - 이 상태로 변경되면 `startAt`, `endAt`는 null로 정리된다.
5. 근태 승인
   - 근태는 상태가 존재할 때만 승인할 수 있다.
   - 같은 날짜의 근태는 당일 승인할 수 없다.
   - 이미 승인된 근태는 다시 승인할 수 없다.
   - 승인 시 기록되는 값: `approvedBy`, `approvedAt`
6. 근태 수정
   - 부서 관리자는 부서 내 직원의 근태를 수정할 수 있다.
   - 이미 승인된 근태는 수정할 수 없다.
   - 수정 시 필수값: `editAt`, `editReason`, `editedBy`, `status`
   - 수정 사유는 공백일 수 없다.
   - 수정할 내용이 실제로 존재해야 한다.
   - 시간 변경 또는 상태 변경이 있어야 한다.
   - 수정 가능 항목: `출근 시각`, `퇴근 시각`, `일일 근태 상태`
   - 근태가 수정되면 아래 정보를 남긴다: (수정자) `editedBy`, (수정일시) `editedAt`, (수정사유) `editReason`

## _Approval_
- Approval은 하나의 Draft에 대한 결재 흐름을 관리하는 애그리거트
- 결재선(Approver)을 보유하며, 상신 전/상신 후/진행 중/완료/반려 상태를 관리한다.

**속성**

| 필드명                  | 비고          | 설명              |
|----------------------|-------------|-----------------|
| `id `                | int         |                 |
| `draft_id `          | int         | draft.id        |
| `status `            | varchar(10) | Enum, 결재 상태     |
| `approvers `         | List        | Approver.결재자 목록 |
| `created_at        ` | datetime    |                 |  
| `updated_at        ` | datetime    |                 |   

#### **ApprovalStatus**
_Enum_
**상수**
- `UNSUBMITTED` : 임시저장 상태, 미상신
- `WAITING` : 결재대기
- `IN_PROGRESS` : 결재진행중
- `APPROVED` : 결재완료
- `REJECTED` : 결재반려

**행위**
- `createDraft()` : 임시저장된 기안서대상 임시 승인정보 엔티티 생성.
- `createSubmitted()` : 기안 상신한 기안서대상 승인정보 엔티티 생성.
- `approve()` : 해당 차례의 결재자의 승인
- `reject()` : 해당 차례의 결재자의 반려처리

**규칙**
1. 엔티티 생성 규칙
   - Approval은 반드시 하나의 Draft와 1:1로 연결된다.
   - 생성 시 상태는 항상 WAITING으로 시작한다.
   - 연관
     - Draft생성 시, Approval가 1:1 관계로 같이 생성되며, 결재선에 따라 Approver이 1:N관계로 생성된다.
2. 결재자 추가 
   - 결재자 추가 규칙 (addApprovers)
   - 동일한 Emp는 중복으로 결재자에 등록될 수 없다.
   - 동일한 결재 순서(order)는 중복될 수 없다.
   - 결재자는 최소 1명 이상 등록되어야 정상적인 결재 진행이 가능하다.
   - 결재자 추가는 결재 진행 전에만 이루어져야 한다
3. 결재상태 규칙
   - 상태 전이 흐름
     ```
     (UNSUBMITTED -> ) WAITING -> IN_PROGRESS -> APPROVED
                                  \-> REJECTED
     ```
4. 생성 : 엔티티 생성시 기안 제출전후에 따라 생성 규칙이 다르다.
   - `createDraft` : 생성 시 상태는 `Unsubmitted` 이다.
   - `createSubmitted` : 생성 시 상태는 `WAITING` 이다
   - 생성시, 필수 파라미터인 결재선은 record인 `List<ApproversParam>`으로 받는다.
   - 결재자 목록(params)은 null일 수 없고 비어 있을 수 없다.
     #### **ApproversParam**
     ```
     - 한 사람은 같은 결재문서에서 한 번만 등장할 수 있다
     - 다만, 본인기안 본인결재의 경우 기안자와 결재자가 동일할 수 있다.
     - 하나의 순서에는 한 명만 배정될 수 있다.
     ```
5. 상신 : 상신은 UNSUBMITTED 상태에서만 가능하다.
   - 결재자가 한 명 이상 존재해야 한다.
   - 상신되면 상태는 WAITING이 된다.
6. 승인 : 승인 후 남은 미처리 결재자가 있으면 상태는 `IN_PROGRESS` 없으면 `APPROVED`
   - 현재 승인 가능한 사람은 미처리 상태(pending)인 결재자 중 가장 낮은 order를 가진 사람이다.
   - 현재 차례의 결재자가 아니면 승인할 수 없다.
   - `REJECT` 또는 `APPROVED` 상태의 결재는 승인할 수 없다.
7. 반려 : 반려 시, 결재 상태는 즉시 `REJECTED`가 된다.
   - 현재 차례(order)인 결재자만 반려를 할 수 있다. 

## _Approvers_

| `approved_At `   | DateTime     | 승인 일시             |
| `reject_Reason ` | text         | 반려 사유             |
| `rejected_At `   | DateTime     | 반려 일시             |
| `created_at `    | datetime     |                   |  
| `updated_at `    | datetime     |                   |   

#### **ApprovalStatus**
_Enum_
**상수**
- `APPROVER` : 결재 승인자
- `COOPERATOR` : 협조 승인자

**규칙**
- 하나의 Approval 안에서 같은 사원(approver_id)은 한 번만 등록될 수 있다.
- 생성규칙
  - approval, role, emp는 null일 수 없다.
  - order은 양수여야 한다. 
- 반려
  - 결재완료 및 반려상태에서 반려할 수 없다.
  - 반려시, 반려이유는 null이거나 빈값이 올 수 없다.
  - 반려시, 반려시각은 null이 될 수 없다.
- 승인
  - 결재완료 및 반려상태에서 승인할 수 없다.
  - 승인시, 반려시각은 null이 될 수 없다.
  
## _Circulation_
- 하나의 Draft를 특정 사원에게 공람 대상으로 지정한 엔티티

**속성**

| 필드명                  | 비고            | 설명       |
|----------------------|---------------|----------|
| `id `                | int           |          |
| `draft_id `          | int           | Draft.id |

**규칙**
- 기안에 대한 공람 설정 시, 해당 엔티티가 생성되며, 생성 직후에는 미열람 상태이다. `read_At = null`

- 공람자가 열람했음을 표시할 시, 열람 처리 시각은 null일 수 있다.

## _Meeting_
- 회의실 예약 정보를 관리한다.
- MeetingRoom와 Emp를 단방향으로 참조한다.

**속성**

| 필드명           | 비고           | 설명                |
|---------------|--------------|-------------------|
| `id          ` | int          |                   |      
| `room_id     ` | int          | meetingRoom.id    |

| `title       ` | varchar(100) | 제목                |

| `start_at    ` | LocalTime    | 시작시각              |      
| `end_at      ` | LocalTime    | 종료시각              |

| `created_at  ` | datetime     |                   |  
| `updated_at  ` | datetime     |                   |   


**행위**
- `static reserve()` : 회의 예약
- `cancel()` : 회의 취소
- `changeReservation()` : 회의시각변경
- `changeTitle()` : 회의명 수정

**규칙**
- 모든 회의는 반드시 하나의 회의실(meetingRoom)에 속한다.
- 모든 회의는 반드시 하나의 예약자(emp)를 가진다.
- 생성
  - 회의 생성 시 회의실, 예약자, 제목, 회의일, 시작시각, 종료시각은 필수다.
  - 종료시각은 시작시각보다 늦어야 한다.
  - 생성 직후 회의는 `isCancel = false`이다.
- 수정
  - 취소된 회의는 정보를 수정할 수 없다.
  - 회의시간 변경 시, 회의일자/시작/종료시각은 필수이며, 종료시간은 시작시간보다 이를 수 없다
  - 회의명 변경 시, 회의 제목은 빈값일 수 없다.
  - 회의실 변경은 취소하고 새 예약을 생성한다
- 취소 
  - 예약한 회의를 취소할 수 있다.
