# Groupware Security 설계 절차

참고한 문서:
- GitBook `auth_api`: https://eunseo-lay.gitbook.io/auth_api
- Spring Security Authorize HTTP Requests: https://docs.spring.io/spring-security/reference/servlet/authorization/authorize-http-requests.html
- Spring Security CORS: https://docs.spring.io/spring-security/reference/servlet/integrations/cors.html
- Spring Security CSRF: https://www.springframework.org/spring-security/reference/6.5/servlet/exploits/csrf.html
- OWASP JWT for Java Cheat Sheet: https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html

## 결론

현재 `groupware`는 React SPA + REST API + hexagonal architecture로 리팩터링 중이므로, 기본 인증은 `loginId/password` 기반 JWT access/refresh token 방식이 가장 자연스럽다.

다만 GitBook 예제처럼 access token과 refresh token을 모두 응답 바디로 내려 프론트 저장소에 보관하는 방식은 구현이 단순하지만 XSS에 약하다. 운영 기준으로는 다음 방식을 권장한다.

- access token: 짧은 만료 시간, React가 `Authorization: Bearer <token>` 헤더로 전송
- refresh token: HttpOnly, Secure, SameSite 쿠키로 전달
- refresh token 원문은 DB나 Redis에 저장하지 않고 해시만 저장
- refresh token은 재발급 때마다 rotation
- 로그아웃 시 refresh token 저장소 삭제
- access token은 짧게 가져가고, 탈취 대응이 필요하면 Redis denylist 또는 token version을 둔다

현재 로컬 코드 상태에서 바로 손봐야 할 지점은 다음이다.

- `adapter/security/SecurityConfig.java`: 현재 모든 요청이 `permitAll`이다.
- `adapter/security/SecurePasswordEncoder.java`: 현재 `encode`는 빈 문자열, `matches`는 항상 `false`를 반환한다.
- `EmpRepository`: 로그인 인증을 위해 `findByLoginId`가 필요하다.
- `application-dev.yml`: JWT 설정은 이미 있으나, 가능하면 `spring.jwt`보다 `app.security.jwt` 같은 애플리케이션 전용 prefix로 옮기는 편이 명확하다.
- `build.gradle.kts`: `jjwt-api`는 `0.11.5`, `jjwt-impl`은 `0.12.6`이라 버전이 섞여 있다. 같은 버전으로 맞춘다.

## 레이어 배치

보안 구현은 헥사고날 경계를 기준으로 나눈다.

| 레이어 | 책임 |
|---|---|
| `domain.empInfo` | `Emp`, `SystemRoleCode`, `PasswordEncoder` 포트 유지. Spring Security 타입을 넣지 않는다. |
| `application.auth` | 로그인, 재발급, 로그아웃 유스케이스. 계정 상태, 비밀번호 검증, refresh token rotation 정책을 담당한다. |
| `application.auth.required` | `RefreshTokenStore`, `TokenIssuer`, 로그인용 사원 조회 포트 등을 둔다. |
| `adapter.security` | `SecurityConfig`, JWT 생성/검증, 인증 필터, `UserDetails` 또는 principal adapter, BCrypt 구현을 둔다. |
| `adapter.web.auth` | `/api/auth/login`, `/api/auth/reissue`, `/api/auth/logout`, `/api/auth/me` 컨트롤러를 둔다. |
| `adapter.persistence` | refresh token Redis adapter, 필요하면 token denylist adapter를 둔다. |

## 1단계: 비밀번호 저장 방식 확정

현재 도메인에는 `PasswordEncoder` 포트가 이미 있다.

구현 방향:
- `SecurePasswordEncoder`는 Spring Security의 `BCryptPasswordEncoder`를 감싸는 adapter로 구현한다.
- 도메인의 `Emp.register`, `changePassword`, `checkPassword`는 지금처럼 포트만 사용한다.
- raw password는 application/service 밖으로 오래 들고 다니지 않는다.

구현 후보:

```java
@Component
public class SecurePasswordEncoder implements PasswordEncoder {
    private final org.springframework.security.crypto.password.PasswordEncoder delegate =
            new BCryptPasswordEncoder();

    @Override
    public String encode(String rawPassword) {
        return delegate.encode(rawPassword);
    }

    @Override
    public boolean matches(String rawPassword, String encodedPassword) {
        return delegate.matches(rawPassword, encodedPassword);
    }
}
```

## 2단계: 로그인 조회 포트 추가

로그인은 `empNo`보다 `loginId`가 자연스럽다. 현재 `Emp`에 `loginId`와 `empPassword`가 있으므로 로그인용 조회가 필요하다.

권장 포트:

```java
public interface EmpRepository extends Repository<Emp, Long> {
    Optional<Emp> findByLoginId(String loginId);
}
```

로그인 시 검증 순서:
1. `loginId`로 사원을 조회한다.
2. 사원이 없으면 401로 응답한다.
3. `EmpStatus.ACTIVE`인지 확인한다.
4. `PasswordEncoder.matches(rawPassword, empPassword)`를 확인한다.
5. 성공하면 access token과 refresh token을 발급한다.

## 3단계: 토큰 모델 결정

access token claim은 최소화한다.

권장 claim:

| Claim | 값 |
|---|---|
| `sub` | `empId` 같은 변경되지 않는 내부 ID |
| `loginId` | 화면 표시나 로그용. 권한 판단의 기준으로 삼지는 않는다. |
| `roles` | `SystemRoleCode` 목록 |
| `jti` | 토큰 고유 ID |
| `iat` | 발급 시각 |
| `exp` | 만료 시각 |

주의:
- `loginId`는 HR에 의해 변경될 수 있으므로 `sub`로 쓰지 않는 편이 낫다.
- 사원명, 이메일, 부서명처럼 자주 바뀌거나 개인정보성이 큰 값은 JWT에 넣지 않는다.
- JWT payload는 서명되어도 암호화된 것이 아니므로 민감정보 저장소로 쓰지 않는다.

만료 시간 권장:

| 토큰 | 권장 |
|---|---|
| access token | 10분에서 30분 |
| refresh token | 7일에서 14일 |

## 4단계: refresh token 저장소 설계

로컬 프로젝트는 Redis 의존성과 설정이 이미 있으므로 refresh token 저장소는 Redis가 맞다.

권장 포트:

```java
public interface RefreshTokenStore {
    void save(long empId, String tokenHash, Duration ttl);
    Optional<RefreshTokenSession> findByHash(String tokenHash);
    void deleteByHash(String tokenHash);
    void deleteAllByEmpId(long empId);
}
```

저장값:

| Key | Value | TTL |
|---|---|---|
| `auth:refresh:{tokenHash}` | `empId`, `issuedAt`, `expiresAt` | refresh token 만료 시간 |
| `auth:emp:{empId}:refresh:{tokenHash}` | marker | refresh token 만료 시간 |

운영 원칙:
- refresh token 원문은 저장하지 않는다.
- SHA-256 같은 digest를 저장한다.
- 재발급 성공 시 기존 refresh token은 삭제하고 새 refresh token을 저장한다.
- 로그아웃 시 현재 refresh token을 삭제한다.
- 전체 로그아웃이 필요하면 `deleteAllByEmpId`를 사용한다.

## 5단계: 인증 API 설계

React용 API는 다음처럼 시작한다.

- POST `./auth/login/param={}` : 로그인
- POST `./auth/reissue/param={}` : access token 재발급
- POST `./auth/logout/param={}` : 로그아웃
- GET `./auth/me/param={}` : 현재 로그인 사용자 조회

요청/응답 원칙:
- 로그인 성공 응답 body에는 access token과 사용자 요약 정보만 둔다.
- refresh token을 쿠키로 쓸 경우 응답 body에 refresh token을 넣지 않는다.
- `GET ./auth/me`는 access token 검증 후 `empId`, `empName`, `roles`, `primaryDept` 정도만 반환한다.

## 6단계: SecurityFilterChain 설정

현재 설정은 모든 요청 허용이므로 다음 순서로 강화한다.

1. `csrf`, `formLogin`, `httpBasic` 정책을 결정한다.
2. REST API는 `SessionCreationPolicy.STATELESS`로 둔다.
3. CORS는 `CorsConfigurationSource`를 명시하고 프론트 origin만 허용한다.
4. JWT 인증 필터를 `UsernamePasswordAuthenticationFilter` 앞에 둔다.
5. 인증 실패는 401, 권한 부족은 403 JSON 응답으로 통일한다.
6. `anyRequest().denyAll()`을 최종 목표로 두고, 개발 중에는 `/api/**.authenticated()`를 거쳐도 된다.

권장 형태:

```java
http
        .cors(cors -> cors.configurationSource(corsConfigurationSource))
        .csrf(csrf -> csrf.disable())
        .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .formLogin(AbstractHttpConfigurer::disable)
        .httpBasic(AbstractHttpConfigurer::disable)
        .authorizeHttpRequests(auth -> auth
                .requestMatchers(HttpMethod.OPTIONS, "/**").permitAll()
                .requestMatchers("/api/auth/login", "/api/auth/reissue").permitAll()
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/api/**").authenticated()
                .anyRequest().denyAll()
        )
        .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class);
```

CSRF 선택 기준:
- access token만 `Authorization` 헤더로 보내는 순수 stateless API라면 CSRF 비활성화가 단순하다.
- refresh token을 HttpOnly cookie로 자동 전송한다면 `/api/auth/reissue`, `/api/auth/logout`은 CSRF 또는 origin 검증을 적용한다.
- GitBook 문서의 설명처럼 Bearer header 방식은 CSRF 위험이 낮지만, 쿠키를 쓰는 순간 다시 CSRF를 고려해야 한다.

## 7단계: JWT 인증 필터

필터 책임:
1. `Authorization` 헤더에서 `Bearer ` 토큰을 꺼낸다.
2. 토큰 서명, 만료, issuer를 검증한다.
3. `sub`에서 `empId`를 얻는다.
4. roles claim을 `GrantedAuthority`로 변환한다.
5. `SecurityContextHolder`에 인증 객체를 넣는다.
6. 토큰이 없으면 통과시키고, 보호 URL에서 Spring Security가 401을 내게 한다.
7. 토큰이 잘못되면 401 JSON으로 응답한다.

권장 principal:

```java
public record GroupwarePrincipal(
        Long empId,
        String loginId,
        Set<SystemRoleCode> roles
) {}
```

권한 문자열:
- `SystemRoleCode.HR`는 `HR`
- `SystemRoleCode.ADMIN`은 `ADMIN`
- `hasAuthority("HR")`처럼 role prefix 없이 통일한다.

`hasRole`을 쓰면 `ROLE_` prefix가 붙으므로, 현재 enum과 맞추려면 `hasAuthority`가 더 단순하다.

## 8단계: HTTP 인가와 유스케이스 인가 분리

보안은 두 겹으로 둔다.

| 위치 | 담당 |
|---|---|
| Spring Security | 인증 여부, 큰 단위 URL 권한 |
| Application service | 본인 여부, 같은 부서 여부, 담당자 여부, aggregate 상태 검증 |

이미 `AuthorizationChecker`와 `docs/권한규칙.md`가 있으므로, HTTP rule은 너무 복잡하게 만들지 않는다.

예시:
- `/api/admin/**`는 `ADMIN`
- `/api/hr/**`는 `HR`
- `/api/franchise/**`는 `FRANCHISE`
- `/api/meeting-rooms/**`의 command API는 `FACILITY`
- 게시글 수정, 쪽지 삭제, 일정 수정처럼 "본인 소유"가 중요한 검증은 application service에서 처리한다.
- `ADMIN`은 대부분 통과하되, 출근/퇴근처럼 반드시 본인 행위여야 하는 기능은 예외로 둔다.

## 9단계: URL 권한 초안

`docs/api-query-map.md`와 `docs/권한규칙.md` 기준으로 시작점은 다음처럼 잡는다.

| URL | 권한 |
|---|---|
| `/api/auth/login` | permitAll |
| `/api/auth/reissue` | permitAll 또는 refresh cookie 검증 |
| `/api/auth/logout` | authenticated |
| `/api/auth/me` | authenticated |
| `/api/users/me/**` | EMPLOYEE |
| `/api/users/new/**`, `/api/users/*/hr/**` | HR |
| `/api/depts/**` command | ADMIN |
| `/api/attendances/depts/**` command | DEPT_MANAGER |
| `/api/franchises/**`, `/api/educations/**`, `/api/franchise-inquiries/**` command | FRANCHISE |
| `/api/meeting-rooms/**` command | FACILITY |
| `/api/boards/**`, `/api/messages/**`, `/api/chatrooms/**`, `/api/schedules/**` | authenticated, 세부 소유권은 application service |

조회 API는 넓게 `authenticated`로 시작하고, 민감한 HR/통계/관리 조회만 role을 좁힌다.

## 10단계: CORS 설정

React와 API 서버가 origin이 다르면 CORS가 필요하다.

권장:
- allowed origin은 `app.frontend.base-url`만 사용한다.
- `*`와 credentials 조합은 피한다.
- 허용 method는 실제 사용하는 method만 둔다.
- 허용 header는 `Authorization`, `Content-Type`, `X-CSRF-TOKEN` 정도로 시작한다.
- refresh cookie를 쓴다면 `allowCredentials(true)`가 필요하다.

예시:

```java
@Bean
CorsConfigurationSource corsConfigurationSource(AppSecurityProperties props) {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(List.of(props.frontendBaseUrl()));
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type", "X-CSRF-TOKEN"));
    config.setAllowCredentials(true);

    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}
```

## 11단계: 프론트 저장 전략

권장 운영안:
- access token은 React 메모리에 보관한다.
- 새로고침 시 `/api/auth/reissue`로 access token을 다시 받는다.
- refresh token은 HttpOnly cookie로만 이동한다.
- axios interceptor는 access token을 `Authorization` 헤더에 붙인다.
- 401이면 한 번만 reissue를 시도하고, 실패하면 로그인 화면으로 보낸다.

간단 구현안:
- access/refresh token을 둘 다 response body로 받고 sessionStorage에 저장한다.
- 모든 API는 `Authorization` 헤더로 보낸다.
- CSRF는 disable한다.
- 이 방식은 구현은 쉽지만 XSS에 취약하므로 운영 전에는 강화한다.

## 12단계: 테스트 절차

최소 테스트:
- `SecurePasswordEncoderTest`: encode 결과가 raw와 다르고, matches가 성공/실패를 구분하는지 확인
- `AuthServiceTest`: 로그인 성공, 비밀번호 실패, 존재하지 않는 사용자, 비활성 사용자 검증
- `JwtTokenProviderTest`: 생성, 만료, 잘못된 서명, roles claim 검증
- `RefreshTokenStoreTest`: 저장, TTL, rotation, logout 삭제 검증
- `JwtAuthenticationFilterTest`: 정상 토큰이면 SecurityContext 생성, 잘못된 토큰이면 401
- `SecurityConfigTest`: 공개 API 200, 인증 없는 보호 API 401, 권한 부족 403, 권한 충족 200
- `CorsTest`: 프론트 origin preflight가 통과하는지 확인

MockMvc 기준:
- 인증 없는 `GET /api/auth/me`는 401
- `EMPLOYEE`가 HR API를 호출하면 403
- `HR`이 HR API를 호출하면 200 또는 유스케이스 결과
- 로그아웃 후 같은 refresh token으로 재발급하면 401

## 구현 순서 체크리스트

1. `jjwt` 버전을 하나로 맞춘다.
2. JWT 설정 prefix를 `app.security.jwt`로 정리하고 `@ConfigurationProperties`를 만든다.
3. `SecurePasswordEncoder`를 BCrypt adapter로 구현한다.
4. `EmpRepository.findByLoginId`를 추가한다.
5. `application.auth` 패키지에 `AuthenticationManagement`, `AuthService`, DTO를 만든다.
6. `RefreshTokenStore` 포트와 Redis adapter를 만든다.
7. `JwtTokenProvider`를 만든다.
8. `JwtAuthenticationFilter`를 만든다.
9. `GroupwarePrincipal`과 authority 변환 규칙을 만든다.
10. `SecurityConfig`를 stateless JWT 방식으로 바꾼다.
11. `AuthController`를 만든다.
12. `docs/권한규칙.md`를 기준으로 URL 권한표를 `SecurityConfig`에 반영한다.
13. 인증/인가 테스트를 추가한다.
14. React axios interceptor와 reissue 흐름을 붙인다.

## 나중에 확장할 것

- 비밀번호 실패 횟수 제한과 계정 잠금
- 관리자 강제 로그아웃
- refresh token 기기별 세션 목록
- 비밀번호 변경 시 전체 refresh token 폐기
- access token denylist 또는 `tokenVersion`
- 감사 로그: 로그인 성공/실패, 권한 거절, 관리자 작업
- OAuth2/SSO는 별도 `SecurityFilterChain`이나 auth provider로 추가
